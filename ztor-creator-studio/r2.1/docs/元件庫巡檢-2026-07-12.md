# ds-components 元件庫瘦身提案

普查範圍：`ds-components/*.css` 扣除 `_tokens.css`、`fonts.css`、墓碑檔（`navigation-menu.css`、`status-axes.css`）共 68 支。消費頁數統計對象：33 支業務 HTML（扣 `design-system.html` 展示頁）＋ `partials/*.js`（10 支）＋ `js/*.js`（11 支，含模板字串內的 class 名）。

統計方式：每支 CSS 先抓全部頂層 class，依 BEM 字首（`--modifier` / `__element`）分組成「家族」，同一支檔案內不同家族分開計消費頁數（例如 `badge.css` 的 `.badge` 家族與 `.ztor-badge` 家族分開統計）。比對時用完整 class 字串比對（含 modifier/element 全名），避免子字串誤判（例如 `.badge` 不會誤吃到 `.ztor-badge`，`.variant-cell--total` 這種「只有 modifier、沒有裸字族名」的家族也能正確算到消費頁）。

**一行結論**：退場 3 支整檔（另有 4 處檔內死碼建議一併清）／合併候選 9 組（3 組建議實際合併或遷移、1 組部分遷移＋觀察、5 組審過＋建議維持現狀）／保留核心 45 支／保留單頁 17 支。

## 二、合併候選（9 組）

### 建議實際合併／遷移（3 組）

#### 1. `product-list.css` vs `project-list.css` — 建議留 product-list，project-list 併入

| | anatomy | 消費頁 |
|---|---|---|
| `product-list.css` | `.product-list` 為 borderless table，`--auctions/--bundles/--eshop/--orders/--pickup` 5 個型態 modifier，5 欄 grid | 6（creators／e-shop／events／orders／pickup.html＋devtools.js） |
| `project-list.css` | `.project-list`，結構幾乎一致（同樣 borderless table、`__head`/`__row` grid pattern），4 欄 grid，無型態 modifier | 2（projects.html＋devtools.js，僅此一頁） |

兩支檔案開頭註解幾乎逐字相同「borderless table list，沿用既有 color/type/spacing/radius/motion tokens」，明顯是同一套 pattern 被複製了一份給 projects 頁用，只是欄位數與欄寬不同。product-list 已經用 modifier 撐 5 種型態，多加一個 `--projects` modifier 是同一套機制的自然延伸。

**建議**：留 `product-list.css`，把 `project-list` 的 4 欄版面收成 `.product-list--projects` modifier，projects.html 改用 `product-list` 命名。工程量：**小**（只影響 projects.html 一頁 + `js/devtools.js` 裡的 demo 資料 stub，欄位語意需要對一次但結構相同）。

#### 2. `chart.css` 內建 `.segmented` vs `segmented.css` — 建議刪掉 chart.css 裡那份，改用 segmented.css

`chart.css` 裡自己刻了一份 `.segmented` / `.segmented__item` / `.segmented__item--active`（第 235–260 行），註解寫著「Generic enough to promote to its own component later」——但 `segmented.css` 早就是獨立元件了，只是子元素命名不同（`__btn` 而非 `__item`）。

- `chart.css` 版（`__item`）：只有 earnings.html 的圖表週期切換（D/W/M）在用，1 頁 + js/chart.js。
- `segmented.css` 版（`__btn`）：7 個消費者（create-auction／create-product／events／order-detail／product-detail.html＋partials/restock-modal.js＋partials/vip-card.js）。

**建議**：刪掉 `chart.css` 裡那份重複定義，earnings.html 的圖表週期切換改用 `segmented__btn` 命名接軌 `segmented.css`。工程量：**小**（只動 earnings.html 一頁 + js/chart.js 的 class 名，視覺效果本來就幾乎一樣，兩份 CSS 的 padding/radius 略有差異需挑一個）。

#### 3. `radio-card.css` vs `selection-card.css` — 建議刪掉 radio-card.css，改用 selection-card

`selection-card.css` 檔頭註解直接寫明：「Merged from project-level `.choice-card`（wizard radio cards）and `.appearance-card`（settings theme picker）on 2026-05-26」，並列出兩種標準組合，第一種就是「Wizard radio card: title + sub + tag（no swatch）」——這正是 `radio-card.css` 現在還在做的事。也就是說 selection-card 早就把「精靈流程二選一卡片」納為己用範例了，`radio-card.css` 是當年沒跟著遷移完的殘留。

- `radio-card.css`：搭在 `.segmented.radio-cards` 之上（疊加 modifier + 三個文字 class），**只剩 create-product.html 一頁在用**（VIP 素材模式 text/image、規格 single/multiple 兩組二選一切換）。`bundle-detail.html`／`create-bundle.html` 原本也用過，但頁面內註解明寫「庫存版本：用 selection-card，非 radio-card」——即該頁已經手動遷移過。
- `selection-card.css`：9 頁在用（bundle-detail／create-auction／create-bundle／create-event／create-product／create-project／register-ip／settings.html＋devtools.js）。

**建議**：刪掉 `radio-card.css`，create-product.html 那兩組二選一切換改用 `.selection-grid` / `.selection-card`（無 swatch 版）。工程量：**小**（僅 1 頁、2 組切換元件，且 codebase 裡其他頁已經示範過怎麼遷移）。

### 部分遷移／待觀察（1 組）

#### 4. 預覽四支：`preview-card.css` × `event-preview-card.css` × `preview-column.css` × `preview-panel.css`

| 檔案 | 角色 | 消費頁 |

**建議**：
- create-event.html 遷移到 `preview-column` + `preview-card` 殼，但活動卡本身欄位（海報/類型標籤/日期場地/票種）跟商品卡差異大，`event-preview-card.css` 的內容本體建議保留，只是外層容器從 `.preview-panel` 換成 `.preview-split`/`.preview-col`。工程量：**中**（1 頁，但要重新排版 sticky 欄位 + 確認活動卡在窄版兩欄裡的斷點）。

- `.main`／`.wizard` 這兩個核心版面 class 放在 `preview-panel.css` 裡只是歸檔位置奇怪（分別是全站 `<main>` 位移動畫、精靈外殼），不是重複定義，不用動，但如果之後要拆檔可以順手搬到更合語意的檔案。

## 附註：普查方法的兩個修正

第一版用「家族根字直接比對」會誤判兩種情況，第二版改成「家族內所有 class 全名比對取聯集」修正：

1. **只有 modifier／element、沒有裸字族名的家族會被誤判成 0 消費**：例如 `variant-builder.css` 的 `.variant-cell--total`／`.variant-row__remove`，CSS 裡從沒定義過裸的 `.variant-cell`／`.variant-row`，第一版比對「variant-cell」這個根字時因為後面接的是 `--`／`__` 而配不到邊界，誤判 0 消費；改成比對完整 class 名後，兩者其實都在 create-product.html 有在用。`store-settings.css` 的 `.ss-band__*` 也是同一種誤判。
2. **複合選擇器會誤判成「這支檔案自己的家族」**：例如 `chart.css` 裡的 `.card.chart-card`、`restock-modal.css` 裡的 `.input.restock-line__qty`、`message-modal.css`/`product-post-modal.css` 裡的 `.payout-dialog.msg-dialog`/`.payout-dialog.npp-dialog`——這些是「疊加在別支檔案既有 class 上加一點客製樣式」，不是該檔案自己擁有的新元件家族，統計時已排除，不算進以上的家族清單。
