# Input 高度盤點 — 2026-07-21

任務：盤點 r2.1 全站「文字輸入類控件」的實際渲染高度，判斷是否需要開 STYLE-DECISIONS 裁決題。
範圍：`ds-components/*.css` 全部 62 支 + `shared.css` + 39 個產品頁的頁內 `<style>` + `partials/*.js` / `js/*.js` 注入的 markup。
不修改任何檔案，本檔案為唯一產出。

---

## 1. 基礎事實：line-height 從哪裡來

全站 `<body>` 的字級／行高只有一處宣告，所有頁面都靠 `shared.css` 這條繼承下來（`design-system.html` 另外用
token 變數寫了一份等價規則，數值相同）：

- `shared.css:77-84`
  ```
  body {
    font-family: var(--font-body);
    font-size: var(--fs-14);        /* = 14px，_tokens.css:252 */
    line-height: var(--lh-relaxed); /* = 1.5，_tokens.css:283 */
  }
  ```
- `.input` / `.textarea` / `.select`（`ds-components/input.css:10-28`）本身**沒有宣告 `line-height`**，所以撐開計算一律用繼承來的 `1.5`，不是瀏覽器預設的 `normal`。
- 凡是元件把 `font-size` 改掉、但沒有另外宣告 `line-height` 的，撐開公式一律是 `font-size × 1.5`（因為還是繼承 body 的 1.5，沒人在中途插一層新的 line-height）。逐項核對過，全站沒有例外。

## 3. `--control-h-*` 消費狀況

`_tokens.css:105-109` 定義 `--control-h-xs:28 / -sm:36 / -md:44 / -lg:52 / -xl:60`，`.ztor-btn` 的尺寸註解（`button.css:32`）寫「共用 control-size 級（與 input 同高度 token，2026-06-25）」。

實際 grep 全站，這套 token 目前的消費者是：

- `.btn`（產品頁實際用的按鈕，236+ 處）：預設 `height:var(--control-h-sm)` = **36px**（`button.css:124`），`--lg` 用 44、`--sm` 用 28。
- `.ztor-btn`（design-system.html 展示用的 canonical 按鈕）：預設 `height:var(--control-h-md)` = **44px**（`button.css:23`），四個尺寸變體對齊全部 5 階。
- `.field-pill` / `.tag-input__field`：`min-height:var(--control-h-md)` = **44px**。
- `.owner-lookup__avatar`：`width/height:var(--control-h-sm)` = 36px——但這是頭像方塊，不是輸入框。

**沒有任何 `.input` / `.textarea` / `.select`（基準版）直接引用 `--control-h-*`。** 基準 `.input` 是靠 padding+font-size 撐出來的 39px，這個數字不落在 28/36/44/52/60 任何一階上——比 `-sm`(36) 高 3px、比 `-md`(44) 矮 5px，是懸空值。

所以「同尺寸的 input 與 button 等高」這句註解目前**只對 field-pill / tag-input 成立**（44px，剛好等於 `.ztor-btn` 預設高和 `.btn--lg`），但對全站最大宗的組合——一顆裸 `.input`（39px）配一顆預設 `.btn`（36px）——是**不成立的**，兩者差 3px，文件與實作不符。這應該是本次盤點裡最值得裁決的一條。

## 4. 意外發現：Request Payout 金額框的「70px hero」是死代碼，實際仍是 39px（regression，不只是高度分岔）

追高度來源時發現一個比「分岔」更嚴重的問題，一併回報：

- `ds-components/payout-modal.css:184-186` 的 tombstone 註解宣稱：`.payout-amount-wrap` / `.payout-amount-prefix` / `.input.payout-amount-input` 已於 2026-07-11 收斂進 `.amount-field.amount-field--hero`，「唯一 consumer `partials/payout-request-modal.js` 已改用新結構」，舊規則已從檔案移除。
- `design-system.html:5017-5018` 也重複這個說法：「這些 class 已無任何 consumer，規則也已從 payout-modal.css 移除」。
- 但實測 `partials/payout-request-modal.js`（檔案 mtime 2026-07-15，比宣稱的遷移日期更新）第 54-60 行**仍然是舊 markup**：`<label class="payout-field">` 包 `<span class="payout-amount-wrap"><span class="payout-amount-prefix">$</span><input class="input payout-amount-input" ...>`，完全沒有改用 `.amount-field.amount-field--hero`。
- 全站 grep `.payout-field` / `.payout-amount-*` 的 CSS 定義：`ds-components/*.css` 裡**一條規則都沒有**（已被刪除，符合 tombstone 的說法），但這些 class 仍被四個活著的 partial 引用：`partials/payout-request-modal.js`（7 處）、`partials/manual-entry-modal.js`（8+ 處）、`partials/restock-modal.js`（多處）、`partials/pickup-session-modal.js`（17 處）。只有 `creators.html` 是真的轉乾淨了（0 處）。

**實際後果**：earnings.html 的「Request payout」彈窗、Manual entry 彈窗、Restock 彈窗，裡面所有標成 `.payout-field` / `.payout-field__label` / `.payout-field__hint` 的欄位標籤、提示文字，現在**沒有任何 CSS 套用**（class 存在但規則已刪光）；金額輸入框因為外層 `.payout-amount-wrap`/`.payout-amount-prefix` 沒有定位樣式、`$` 前綴不會對齊，且 `<input class="input payout-amount-input">` 因為 `.payout-amount-input` 本身也沒規則，退回裸 `.input` 的 **39px**，不是文件宣稱的 **70px hero**。

這不是「高度分岔」的裁決題，是一個「文件說已完成的遷移、程式碼其實沒做」的落地缺口，建議獨立通報、不建議跟下面的裁決題混在一起處理。

## 5. 建議的裁決題目

以下只描述題目，不代裁決：

1. **裸 `.input`（39px）要不要正式掛上 `--control-h-*` 刻度**，讓它跟同排最常見的 `.btn`（36px）真正等高？如果掛，掛哪一階（36 會讓 padding 變窄；還是另開一個介於 sm/md 之間的階、或乾脆放棄「input 跟 button 等高」這句文件承諾)？
2. **`.picker__search-input`（41.5px）、`.app-sidebar__search-input`（37.5px）、`.app-topbar__search-input`（35.5px）這三個搜尋框，要不要收斂成同一個高度／同一套 padding 公式？** 目前看不出三者分別要矮一點或高一點的產品理由，像是三次各自手調的殘留。
3. **`_tokens.css:105-109` 的 `--control-h-*` 註解「同尺寸的 input 與 button 等高」要不要改寫**，把「目前只有 field-pill/tag-input 這兩個容器型元件在用」的事實寫清楚，避免下一個人以為裸 `.input` 也在用。
