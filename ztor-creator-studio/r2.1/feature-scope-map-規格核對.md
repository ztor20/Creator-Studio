# Feature Scope Map × 設計規格書 · 核對報告

核對日期 2026-06-30。把 [feature-scope-map.md](feature-scope-map.md) 的 107 個功能點，逐一對照 `documents/` 設計規格書（5.1.5 電子商店群、5.1.5.3 訂單群、5.1.8 收入群），判斷「對不對得上」。

## 一句話結論

創作者後台三塊（S 商店 50／O 訂單 23／E 收入 24，共 97 項）幾乎都能在規格找到對應落點，沒有「規格完全沒寫」的孤兒項——唯一的 ✗（E19 載入更多）也只是規格不約束的呈現細節。真正要處理的不是「對不上」，是三類問題：**結構性缺口**（POPCORN 付款、E 收入分類的層級切法）、**版本切割與規格狀態對不齊**（tier 標 TBD/Next 但規格其實已寫定）、**少數命名與筆誤**。買家店面 B（10 項）整段在 Creator Studio 規格書範圍外——規格書只寫創作者後台，沒有任何買家端頁面規格。

## A. 最該先決定的事（要上游／產品拍板，原型不能自己定案）

### A1. POPCORN（爆米花）付款與定價，整個沒有規格 — 最重大
- 影響：**S30 爆米花價（POPCORN price）** ＋ 買家端 **B06 爆米花付款（Pay by POPCORN）**，是同一套機制。
- 事實：全 `documents/` 搜尋 `POPCORN`／`爆米花`，**零命中**。規格 5.1.5.2 §4.1 F3.2 定價只有「價格／原價／成本價」。
- Scope 自己也標 net-new／gap。這是一個被列進 Phase 1（S30 🟢）、但設計規格書裡毫無定義的產品規則缺口。要先在 `documents/` 補 POPCORN 的定價與付款規則，下游才有依據。

### A2. E 收入「來源分類」該是分頁 tab，還是篩選維度？ — 結構性
- 影響：**E12–E18**（電子商店／電子票券／IP 版稅／授權／串流版稅／專案支持／提款退款）。
- 衝突：Scope 把這些當成「交易明細」下的**並列分頁 tab**；規格 5.1.8 只設五個分頁（Overview／Transactions／Breakdown／Payouts／Tax Documents），這些來源其實是 `0-設計規格書.md §7.3 收入與交易分類`（標「暫存」、未定稿）的**一條獨立分類軸**，套用在 F5 來源占比與 F7 交易明細的分類欄。
- 若照 Scope 實作成多 tab，會與規格的分頁架構（F2）直接衝突。要先對齊：來源分類是「篩選/分類」還是「分頁」。§7.3 標「暫存」也正是 Scope 很多 ⚪ TBD 的來源。

### A3. 拍賣線在 S pillar 只盤了一個入口，規格卻很完整 — 顆粒度不對等
- 事實：Products（S09–S39）、Bundles（S40–S50）都逐項盤點；Auctions 只有 **S11 一個建立入口**（標 deferred）。但規格已有完整拍賣線：`5.1.5.10 建立拍賣`（F1–F6）、`5.1.5 F4 競標清單`、`F6 拍賣即時橫條`、拍賣狀態機、`5.1.5.8 拍賣商品細節頁`。
- 結論：Scope 對拍賣是「整體延後＝不展開編號」。若日後拍賣轉 Phase，S 編號要大幅補列。現況不算錯，但要知道 scope map 沒有反映規格已寫好的拍賣細節。

### A4. 數位商品在 Phase 1 到底做不做？ — 方向衝突
- 影響：**O22 數位下載**（標 🔵 Next、digital goods deferred）。
- 衝突：規格 `5.1.5.3.1 訂單詳情 §2.5` 已把數位交付當成常態履約分支寫定，§2.6 也寫了數位不回補，訂單管理 F4 也明寫「數位商品處理交付／存取」。規格把它當已定義常態，Scope 卻整體延後。

### A5. 退款／爭議的版本切割分散在三種 tier
- 影響：**O04（⚪TBD）、O09（⚪TBD）、O18（⚪TBD）、O23（🔵Next）**。
- 規格：退款/爭議在訂單管理 F1（KPI）、F2（篩選 tab）、訂單詳情 §2.2／§2.6 都已正式寫入，並明確界定「v1 不開放主動退款（disabled）、爭議保留」（§7.3／D041）。
- 建議對齊：**顯示層**（退款 KPI、篩選 tab、爭議）＝ Phase 1；**主動退款動作** ＝ deferred。目前同一組概念被切成三種 tier。

## C. 系統性現象：Scope 的「ahead／TBD／Next」與規格狀態對不上

Scope 的 Build 欄把不少項目標 `✅⬆ ahead`（原型已建、規格尚未涵蓋），但實測規格**多半已涵蓋**：

- **E08 申請提款、E20 手動補登、E23 提款**：標 🔵 Next，但規格 F8／F10／`5.1.8.1`／`5.1.8.2` 都有完整到欄位與檢核的正式規格。
- **E13–E17、E22、E24**：標 ⚪ TBD，但規格 §7.3 分類、F9 稅務、F11/F12 瀑布都已有口徑。
- **O04/O09、S24、S11**：標 TBD/ahead，但規格已寫入。

成因：scope map 是「相對兩份 prototype 的落地狀態」盤點（見其頁尾），不是相對 `documents/` 規格盤的。所以它的 built/ahead 講的是「原型做了沒」，與「規格寫了沒」是兩件事。**核對時要把這兩個維度分開看**——這也是 mapping 最大的價值：scope map ≠ 規格覆蓋表。

## E. 買家店面 B（B01–B10）：規格書範圍外

- 事實：`documents/` 搜尋 購物車／結帳／商品詳情頁／PDP／貨到付款／Apple Pay／爆米花付款／storefront／checkout，**全部 0 命中**；`0-設計規格書.md §3.2 sitemap` 的頁面樹也沒有買家端節點。
- Scope 自己標：「from Phase 1 handoff (not in 功能点.md)」。
- 結論：Creator Studio 是創作者後台，**買家店面不在這份規格書的範圍**。買家端在規格裡唯一的影子是「See as fan 預覽」用的 `fan-store` 共用元件（§6.7），那是創作者預覽自己店面，不是完整的買家購物／結帳流程規格。
- 待辦：B01–B10 若要做（多數標 🟢 Phase 1），需要另一份買家端規格來源（或併入本規格書並擴 sitemap），否則這 10 項沒有設計合約可依。
